# 意思決定: 自動化するか手動で割り切るか

## 背景

k3s クラスターに Traefik / ArgoCD をインストールする際、cloud-init・Ansible・手動 Helm のどれで行うかを判断した。

## 判断基準

以下の観点で「自動化の投資対効果」を評価する。

### 1. 作業の頻度

- **1回きり or 極めて稀** → 手動で割り切る
- **繰り返し発生する** → 自動化の価値が高い

### 2. 自動化の導入コスト

- 既存の仕組み（Ansible, CI/CD 等）がある → 低コストで乗れる
- ゼロから導入が必要 → コストが作業量に見合うか考える

### 3. 失敗時のリカバリ

- 手動なら調査・リトライが容易
- cloud-init のように「VM作成時に1回だけ」の仕組みは失敗時の対応が困難

### 4. その先に自動化があるか

- 手動ステップの先に GitOps 等の自動化基盤がある場合、ブートストラップは手動で割り切れる
- 手動ステップが連鎖して増え続ける場合は自動化すべき

## 今回の判断

| 方法 | 評価 | 理由 |
|------|------|------|
| cloud-init | 不向き | OS 初期設定向き。Helm install の失敗時に調査困難 |
| Ansible | 最適だが見送り | リポジトリに土台がない。1回きりの作業に対して導入コストが大きい |
| Terraform provisioner | 不向き | apply のたびに意図しない再実行のリスク。公式も最終手段と推奨 |
| **手動 Helm** | **採用** | 1回きりのブートストラップ。ArgoCD 以降は GitOps で自動化される |

## 原則

> **「自動化のゴール手前の最後の手動ステップ」は手動で割り切ってよい。**

ArgoCD が動き始めれば以降のワークロードは GitOps で自動管理される。その ArgoCD 自体を立てるための1回きりの作業を自動化する投資対効果は低い。

## 再検討のトリガー

- クラスターの再構築が**2回以上**発生したら Ansible 化を検討
- ノード数が増えて手動作業の対象が増えたら自動化の価値が上がる
