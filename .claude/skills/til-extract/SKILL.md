# TIL抽出

会話やファイルからQ&A・学び・ハマりポイントを抽出し、`knowledge/til/` にTILとして保存する。

**手動発動専用**: 会話の終わりや、学びがあった時に `/til-extract` で発動。

## 使い方

```
/til-extract                          # 会話から抽出（従来）
/til-extract path/to/file.md          # 指定ファイルから抽出
/til-extract ideas/deep-dive/*.md     # glob指定で複数ファイルから抽出
```

## 入力モード判定

| 引数 | モード | ソース |
|------|--------|--------|
| なし | 会話モード | 現在のセッションの会話 |
| ファイルパス | ファイルモード | 指定ファイルの内容 |

## 実行手順

### 0. 入力ソースの決定

- **引数なし**: 会話モード → ステップ1へ
- **引数あり**: ファイルモード → ステップ1Bへ

### 1. 会話の振り返り（会話モード）

直前の会話（現在のセッション）を振り返り、以下のシグナルでTILを探す：

**検出シグナル**:
- 質問→回答のQ&Aパターン（ユーザーが「これってどういうこと？」等と質問し、回答を得た場面）
- 「知らなかった」「勉強になった」「初めて知った」「へー」
- 「ハマった」「エラーが出た」「壊れた」「罠だった」
- deep-diveやWebSearchで新知見を得た場面
- 「なるほど」「そういうことか」
- 設定変更やツール導入で試行錯誤した過程

### 1B. ファイルからの抽出（ファイルモード）

指定されたファイルを読み込み、TILを抽出する。

**ファイルモードの抽出基準（会話モードより厳しめ）**:

ファイルは会話と違い、整理済みの文章であることが多い。全文をTIL化しても意味がないため、以下に該当するものだけを抽出する：

- **意外性のある事実**: 一般的な認識と異なる知見、「実は〜だった」系
- **具体的な数値・制限・仕様**: 曖昧でなく、ピンポイントで役立つ情報
- **落とし穴・注意点**: 知らないとハマるポイント
- **比較・選定の判断基準**: なぜAではなくBなのか、具体的な理由

**抽出しないもの**:
- ファイルの概要・まとめ（それはファイル自体を読めばいい）
- 一般的な説明・定義（教科書に書いてある程度のこと）
- 文脈がないと意味が薄い情報

**source**: ファイルモードの場合、frontmatterの `source` をファイルパスにする（例: `source: ideas/deep-dive/20260221-xxx.md`）

### 2. type自動判定

各TILのtypeを以下の基準で判定する：

| type | 検出パターン |
|------|------------|
| troubleshoot | エラー、ハマり、修正、対策の文脈 |
| discovery | 新しく知った、初めて知った、調べた結果 |
| comparison | AよりB、比較、選定、結局Cにした |
| insight | 気づき、考え方の変化、「こう考えるとうまくいく」|

### 3. 出力フォーマット（typeごと）

**troubleshoot型**: エラーメッセージ・コマンドをコードブロックで原文保持

```markdown
---
title: ""
type: troubleshoot
tags: []
created: YYYY-MM-DD
source: session  # or file path (e.g. ideas/deep-dive/xxx.md)
---

`コマンド` を実行したらエラー:

```
$ 実行コマンド
エラーメッセージ原文
```

原因: ...

対策:

```
$ 解決コマンド
```
```

**discovery型**:

```markdown
---
title: ""
type: discovery
tags: []
created: YYYY-MM-DD
source: session
---

知ったこと。背景があれば補足。
```

**comparison型**:

```markdown
---
title: ""
type: comparison
tags: []
created: YYYY-MM-DD
source: session
---

選択肢: A vs B
判断基準: ...
結論: Bを選んだ。理由は...
```

**insight型**:

```markdown
---
title: ""
type: insight
tags: []
created: YYYY-MM-DD
source: session
---

気づき。なぜそう思ったか。
```

### 4. ファイル名・保存先

- **保存先**: `knowledge/til/`
- **ファイル名**: `YYYYMMDD-{slug}.md`
- slugはタイトルから生成したkebab-case（英語、30文字以内）
- 1 TIL = 1ファイル（複数ネタを1ファイルにまとめない）

### 5. 保存・コミット

- `git add knowledge/til/*.md` でまとめてステージング
- コミットメッセージ: `Add TIL: {件数}件（{日付}）`

### 6. 出力報告

抽出結果を以下のフォーマットで報告する：

```
[til-extract] 会話を振り返り中...          # 会話モード
[til-extract] ファイルを読み込み中... (2件) # ファイルモード
[til-extract] {N}件のTILを検出しました

1. [troubleshoot] miseでprocsがインストールできない → knowledge/til/20260222-mise-procs-aqua-registry.md
2. [discovery] Kafka 4.2のShare Groups → knowledge/til/20260222-kafka-42-share-groups.md
3. [insight] 「言う」と「伝わる」の違い → knowledge/til/20260222-saying-vs-conveying.md

保存しました。不要なものがあれば教えてください。
```

## 抽出ルール

- **ユーザー自身の言葉での理解を尊重する**。ユーザーが「要は〜ってことだよね」「つまり〜ということか」と自分の言葉で理解を表現した場合、その表現をTILの「## 自分の理解」セクションとしてそのまま残す。整理された説明に書き直さない。後で読み返した時に「あの時どう理解したか」が分かることに価値がある
- **無理にTILを作らない**。学びがなければ「今回の会話ではTILは見つかりませんでした」と報告
- エラーメッセージ・ログ出力・コマンド出力は**コードブロックで原文保持**（要約しない）
- **ただしシークレット（トークン、パスワード、API キー、署名付きURL、プライベートホスト名等）は必ずリダクトしてから保存する**。`<REDACTED>` や `xxx` に置換し、原文のままコミットしない
- スタックトレースは関連部分を抜粋（全文不要）
- 実行コマンドもセットで残す（コマンドに含まれるシークレットも同様にリダクト）
- 1セッションで複数TILが出てもOK
- 同じ日に同じslugのファイルが既にある場合はslugに連番を付ける（例: `20260222-mise-procs-aqua-registry-2.md`）
- **根拠URLをできる限り含める**。公式ドキュメント、GitHub Issue/Discussion、ブログ記事等、情報の出典を本文末尾に `## 参考` セクションとして記載する。会話中のWebSearch/WebFetchで得たURL、deep-diveのURL、既存のknowledgeファイルパス等が対象。URLが特定できない場合（自身の経験ベース等）は省略可

## 注意事項

- ユーザーの発言をそのまま引用しない（プライバシー配慮）
- article-seedとの違い: TILは「自分が学んだこと」のメモ。記事ネタ（article-seed）は「他人に伝える価値があるもの」。重複してもOK
- 既存TILとの重複チェックは不要（後で手動で整理する前提）
